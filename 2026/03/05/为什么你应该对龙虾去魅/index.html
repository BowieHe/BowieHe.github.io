龙虾体验记：从入门到放弃

发表于2026-03-05|更新于2026-03-05|开源工具技术反思
|总字数:2.2k|阅读时长:6分钟|浏览量:

为什么你应该对龙虾去魅

最近一周，我折腾了一下 OpenClaw（社区里大家叫它”龙虾”）。想写篇文章记录一下使用体验，给正在观望的朋友做个参考。

事先声明：这不是一篇黑稿。我只是想分享一个普通用户的真实经历——从满怀期待，到遇到各种坑，最后理性看待这个过程。

一、为什么想试试龙虾

2026 年初，OpenClaw 突然火了。GitHub 星标从 0 涨到 18 万+，社交媒体上到处都是”AI 员工”、”数字助理”、”钢铁侠贾维斯”之类的说法。

我正好订阅了阿里云的 Coding Plan，想着反正有免费额度，不如试试看。毕竟，谁不想要一个 7×24 小时在线、能处理邮件、管理日程、甚至写代码的 AI 助手呢？

结果折腾了一周，我的感受是：这东西确实有意思，但远没有网上说的那么美好。

二、安装过程比想象复杂

WSL2 的 systemd 问题

我的主力环境是 Windows + WSL2。按照官方文档，先尝试在 WSL2 里安装：

然后卡住了——systemd 启动失败。

明明 WSL 配置里已经开了 systemd，但龙虾就是报 “systemd not enabled”。去 GitHub Issues 搜了一下，发现不少人遇到同样的问题（#27267）。

折腾了两个小时，改配置、重启 WSL、重装，都没用。最后只好放弃 WSL 方案。

macOS 的 Node.js 版本要求

转战 macOS 后，安装顺利了一些，但新问题又来了：Node.js 必须 >= 22。

我本地有项目依赖 Node 18，有的依赖 Node 20。为了龙虾，只能用 nvm 来回切换。对我这种开发者来说不算大事，但我在想：如果是完全不懂技术的普通用户呢？

那些看着教程想尝鲜的朋友，可能连 Node.js 是什么都不知道。让他们处理版本冲突、nvm 切换、权限问题，门槛确实有点高。

一位用户在掘金上吐槽得很形象：”全网都说一天就能搞定，放屁。我搭建的时候，但凡能崩的东西，全崩了。”

三、稳定性是个问题

好不容易安装完成，配好了 Telegram Bot，以为可以开始用了。

结果它每隔十几分钟就 disconnected。

没有错误日志，没有提示，就是突然断开。需要手动运行 openclaw gateway restart 才能恢复。

去社区一查，这不是我一个人的问题。腾讯云社区有位用户分享：他的龙虾 3 天内”偷偷崩了 4671 次”，直到发现定时任务连续 3 天没执行，才意识到出问题了。

算下来平均每分钟重启 1 次还多。这种稳定性，确实让人不敢把重要任务交给它。

四、配置体验不够友好

因为我本地有代理，想着用 Telegram 应该最稳定。毕竟 Telegram Bot API 已经很成熟了。

但实际配置下来，体验并不顺畅。

初始化的时候有几个选项，如果选错了，后果不是”配置失败”，而是整个系统起不来，需要重新安装。比如：

  • 代理格式写错了？崩
  • Webhook 和轮询模式选错？崩
  • Token 权限不够？崩

最麻烦的是，这些问题往往没有清晰的错误提示。你得去翻日志、查 GitHub Issues、去社区问人。

当然，这里也要客观说：我用的是阿里云 Coding Plan 的 Kimi API，不排除是 Coding Plan 的 Kimi 版本有局限。 但无论如何，一个配置错误就导致整个系统崩溃，这种设计确实不够健壮。

五、Token 消耗比预期高很多

如果说安装和稳定性还能忍，那 Token 消耗才是真正让我”冷静”下来的原因。

龙虾的宣传是”开源免费”，但实际上：

软件本身免费，但用起来可不便宜。

腾讯新闻有篇报道，一位用户分享：

  • 一周消耗 1.2 亿 Token，费用 $183（约 ¥1300）
  • 平均每天 ¥185
  • 极端情况下，6 小时就能烧掉 $170

我自己的体验是：仅仅是让龙虾”待机”，每天也要消耗几千万 Token。因为它有”心跳”机制、上下文保持、后台轮询——这些你看不见的地方，都在持续消耗 Token。

有位 Reddit 用户说得挺形象：”我以为开源就是免费，结果一看账单几百美元。后来才明白，这不是工具成本，这是生命体的新陈代谢。”

六、写代码的能力一般

既然龙虾这么折腾，那它的代码能力应该很强吧？

实测下来，一般般。

我试着让它改一个 Python 脚本，结果：

  • 反复修改好几次，Token 哗哗地烧
  • 改完的代码还是有 bug
  • 上下文一长，就开始”失忆”

相比之下，我用 OpenCode 配合内网穿透，在外面也能 vibe coding：

  • 稳定性更好，不会频繁断开
  • 代码质量更高
  • Token 消耗更可控

龙虾的优势在于”多 Agent 协作”、”消息平台集成”，但如果你主要是想写代码，它未必是最好的选择。

七、其他用户也遇到类似问题

我的经历不是个案。搜索过程中，看到不少类似的反馈：

新旧版本冲突
有用户提到：”新旧版本抢端口，装完直接无限重启，你管这叫升级？”

多 Agent 配置复杂
一位用户分享搭建”AI 写作团队”的经历：四个 Bot 协作，结果跨 Agent 通信权限没开，任务分配后其他 Bot 没反应。查日志才发现需要设置 tools.sessions.visibility=all——文档里根本没说清楚。

安全问题
SegmentFault 上有文章指出：ClawHub 扩展市场出现过 14 个恶意”技能”，专门针对加密货币用户。同时有 1800 多个 OpenClaw 实例暴露了 API 密钥、聊天记录等敏感信息。

网络超时误报
有安全研究人员警告：OpenClaw 会把网络超时误判为上下文溢出，触发死循环。

八、龙虾也有它的价值

说了这么多问题，但我并不是要全盘否定 OpenClaw。

在某些场景下，它还是不错的选择：

作为 Manus 的平替
Manus AI 功能强大，但需要邀请码、价格不菲。龙虾是开源免费的，如果你想体验”AI Agent”的概念，它是一个低成本的入门方式。

简单的任务自动化
对于定时发消息、文件整理、基础数据抓取这类简单任务，龙虾是可以胜任的。它支持 50+ 消息平台（Telegram、WhatsApp、Discord、飞书等），这一点确实方便。

学习 Agent 架构
作为开源项目，龙虾的代码是公开的。如果你想了解 AI Agent 的架构设计、多轮对话管理、工具调用机制，它是一个不错的学习材料。

九、总结：理性看待，按需选择

回到标题：为什么你应该对龙虾去魅？

因为它被过度神话了。

社交媒体上到处都是”一天搭建 AI 员工”、”开源免费替代 Manus”、”钢铁侠贾维斯”的宣传。但实际体验下来：

  1. 安装有一定门槛——Node.js 版本、systemd 配置、代理设置，对非技术用户不够友好
  2. 稳定性还有提升空间——频繁重启、disconnected、无声崩溃
  3. 配置需要耐心——一步错可能导致全盘崩，文档还可以更完善
  4. 使用成本要考虑——“开源免费”只是软件本身，Token 消耗可能不低
  5. 代码能力中规中矩——对于纯编程任务，不如专门的 AI 编程工具

龙虾可能适合这样的人：

  • 有一定技术背景，能处理安装配置问题
  • 对稳定性要求不高，能接受偶尔重启
  • 主要需求是任务自动化，而非复杂编程
  • 有 API 预算，或者使用免费模型（如 Gemini Flash）

如果你期望的是：

  • 开箱即用的体验
  • 钢铁侠贾维斯级别的智能助理
  • 稳定可靠的 7×24 小时服务
  • 低成本的代码助手

那可能需要降低预期，或者考虑其他方案。

十、写在最后

OpenClaw 是一个很有潜力的项目，但目前更像是一个技术爱好者的玩具，而不是普通用户的工具。

如果你只是想尝鲜，不妨试试看。但如果你期望它能立刻改变工作流、替代部分工作，我建议先观望，等它更成熟一些。

毕竟，调教 AI 助手的过程，有时候比调教小爱音响还要痛苦。

而我们都值得更好的体验。

本文基于 OpenClaw v2026.3.2 版本的使用体验，部分问题可能在后续版本中得到修复。如果你有不同看法，欢迎交流。
